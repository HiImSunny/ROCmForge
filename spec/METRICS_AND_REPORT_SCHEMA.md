# ROCmForge — Metrics & Report Schema

## Metrics Captured (Real MI300X Data)

Every benchmark job must capture and report these on the actual hardware.

### Raw Captured Metrics (from amd-smi / rocm-smi)
- gpu_utilization_percent (compute + memory if separate)
- power_watts (avg, peak, during kernel)
- temperature_c (edge, junction if available)
- memory_used_mb / total_mb
- clock_sclk_mhz, clock_mclk_mhz (during hot section)
- timestamped timeseries for live sparklines (every 100-500ms during benchmark)

### Derived Efficiency Metrics (Calculated)
For each seed/kernel:

**Memory Bandwidth (vectorAdd style)**:
- bytes_moved = (size * 2) or measured from kernel (read + write)
- time_s = kernel execution time (hipEvent or wall with care)
- achieved_bw_gbs = bytes_moved / time_s / 1e9
- theoretical_peak_gbs = 5300  # MI300X HBM3 example — confirm on instance
- efficiency_percent = (achieved_bw_gbs / theoretical_peak_gbs) * 100

**Compute (tiled matmul style)**:
- flops = 2 * M * N * K (for matmul)
- achieved_tflops = flops / time_s / 1e12
- theoretical_peak_tflops = 163.4 (FP32) or higher for FP16/BF16/FP8 — use correct one
- efficiency_percent = ...

Always log:
- Data type used (FP32, FP16, etc.)
- Launch configuration (block size, grid)
- Any special flags (gfx942, unsafe atomics, etc.)
- Power and thermal envelope during the measurement

### Other Job Metadata
- job_id, seed_name, timestamp_start/end
- hipify_command + version
- hipcc_command + full flags
- vllm_model_used (for agent brains)
- number_of_repair_loops
- agent_decisions_count
- total_wall_time

The Reporter agent must generate a polished Markdown following this structure (or very close).

```markdown

**Date**: {timestamp}

**vLLM Model (agent brain)**: {model}

## Executive Summary
[Shock metric + outcome in 1-2 paragraphs. Example: "vectorAdd ported and optimized autonomously in 47 seconds on real MI300X. Achieved 4.6 TB/s (87% of theoretical 5.3 TB/s peak) with sustained 680W and 94% utilization."]

## Problem & Scope
- Original CUDA source summary
- What the swarm was asked to do
- Limitations acknowledged up front

## Swarm Journey (Step-by-Step with Agent Decisions)
1. **Analysis** (Analyzer agent)
   - Detected kernels, patterns, risks
   - Key output

2. **Porting & Repair** (HIP Porting Specialist)
   - hipify command + result
   - Repair loop log (1..N): error → patch rationale → outcome
   - Final hipcc command that succeeded

3. **Validation** (Validator)
   - Correctness result + numerical closeness (max diff, tolerance)
   - Any behavioral notes

4. **Benchmark & Profiling** (Benchmark & Profiler)
   - Raw timing
   - Live captured metrics table (util, power, temp, clocks)
   - Derived efficiency (with % of peak)
   - Timeseries summary or link to JSON

5. **Optimization** (if run)
   - What was tried
   - Before/after numbers + % improvement

6. **Reporting**
   - Artifacts generated
